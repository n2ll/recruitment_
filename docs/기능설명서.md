# 1. 지원 폼 (`/apply`)

**용도**: 당근·배민 두 채널의 지원자가 폼을 작성하는 단일 입구.

**섹션 6개**:
1. 기본 정보 — 이름·생년월일·전화·거주지(우편번호 검색)
2. 차량 — 자차 여부·면허·차종
3. 희망 근무 지점 — 1지망(필수)·2지망(선택)
4. 희망 근무 시간대 — 평일오전/평일오후/주말오전/주말오후 **복수 선택**
5. 경력 — 선택 입력
6. 추가 확인 사항 — 시작 가능일·본인 명의·지원 경로·마케팅 동의

## 지원경로 (source) 라디오

**당근 / 배민 두 개만**. URL 파라미터로 디폴트 결정:
- `/apply?source=danggeun` → 디폴트 당근
- `/apply?source=baemin` → 디폴트 배민 (배민 SMS 분류 후 안내된 링크가 이걸 씀)
- 파라미터 없으면 **당근 디폴트**

> ⚠️ 페이스북·네이버·해당없음 옵션은 제거됨. 폼은 당근/배민만.

## 자동 필터

3개 모두 충족해야 통과:
1. 자기 차량 **있음**
2. 면허 **1종 보통 / 2종 보통 / 1종 대형**
3. 본인 명의 **문제 없음**

→ 통과: `status='스크리닝 중'` + 시작 멘트(`danggeun_start`) SMS 발송 + 카카오 채널 추가 페이지 노출
→ 탈락: `status='부적합'`

## 배민 채널 특별 처리 — 중복 row 방지

배민 흐름은 다음 순서:
1. 지원자가 SMS 보냄 → triage AI가 배민 지원으로 분류
2. **임시 applicants row 생성** (`status='스크리닝 전'`)
3. apply 폼 URL을 SMS로 안내
4. 지원자가 폼 작성 → `/api/apply`가 같은 전화번호의 임시 row를 **INSERT가 아니라 UPDATE**해 데이터 일관성 유지

## 지점 on/off

`/api/branches`는 active=true 지점만 반환 → **/apply 폼에는 비활성 지점이 안 보임**.
admin 화면(목록·대시보드·매트릭스·모달)은 모든 지점 그대로 노출 (매니저 작업 편의).

## 2-1. 당근마켓구인 / 배달의민족구인

두 채널 모두 같은 화면(`DanggeunView`)을 mode만 바꿔 재사용. 차이는:

| 항목 | 🥕 당근 | 📱 배민 |
|---|---|---|
| 유입 | apply 폼 / 매니저 수동 등록 | 지원자가 SMS 먼저 → AI triage → apply 폼 안내 → 폼 작성 |
| 시작 멘트 자동 발송 | ✅ (`danggeun_start`) | ✅ (`danggeun_start` 공유) |
| ⭐ 추천 받기 버튼 | ✅ | ❌ |
| AI 자동 응대 | ✅ | ✅ |

### 화면 구조
- **좌측 목록**: 해당 source 후보. 이름 / stage 배지 / 전화 / 지점 / 마지막 대화 시각 / 안 읽음
- **우측 대화창**: 선택 후보의 대화. AI 메시지엔 🧠 판단 근거 표시
  - 상단 **[⏸ AI 일시정지]** / **[새로고침]**
  - 단계 progress (클릭 가능) + 통합 체크리스트 (8항목) + 매니저 직접 발송 입력

### 단계 progress (2단계 표시 — 클릭 가능)

```
스크리닝 중 → 스크리닝 완료
```

> 내부적으로는 `screening → onboarding → active` 3단계지만, UI는 2단계로 단순화. `onboarding`/`active` 모두 "스크리닝 완료"로 표기됨.

- 지난 단계: 초록 ✓
- 현재 단계: 노란 강조
- **다른 단계 노드를 클릭하면** 수동으로 그 단계로 이동 (이전 단계 체크리스트 자동 채움)
  - 스크리닝 → 온보딩 클릭: 앱설치 안내 자동 발송 포함
  - 매니저가 별도 확정 처리는 `상태` 컬럼에서 `확정인력`/`대기자`로 변경 (자동 발송 없음)

### ⏸ AI 일시정지 / ▶ 재개
- 우상단 **[⏸ AI 일시정지]** 클릭 → stage='paused' + Slack '매니저 인계 필요' 알림
- 빨간 배너에 **[▶ AI 응답 재개]** 버튼 → 직전 stage로 복귀
- 자동 paused(정책 질문·부분 가능일 등)와 동일한 메커니즘

### 통합 체크리스트 (8항목, 단일 위젯)

**스크리닝 7항목** (시작일 묻기 X — "에이전트 질문에 긍정 = 근무 확정"이 절대 아니라는 점을 반영):
1. 자차_재확인 (폼 거짓 케이스 대비)
2. 프로모션_종료가능성_안내
3. 정산주기_안내
4. 공휴일_업무여부_확인
5. 본인명의_정산_문제없음
6. 업무시간_체계_이해
7. 지원자_질문_해소

→ 시작 멘트에 안내 묶음 포함되므로 2·3·6번은 자동 true. 평일 슬롯이면 4번도 자동 true.

**온보딩 1항목**:
8. 배민_아이디_수신 — 지원자가 답장에 배민 커넥트 ID(영문/숫자) 보냈는가

> ⚠️ **차량번호 수집은 제거됨** — 배민 아이디만 받으면 AI 응대 완료. 만남장소·차량번호 등은 매니저가 직접 처리.

### "확정 뉘앙스 절대 금지" 룰
AI는 정보를 받아도 그게 곧 근무 확정/배정을 의미하지 않는다는 점을 명확히 함. "근무 진행", "곧 시작합니다" 류의 확정 멘트 금지. 매니저가 별도로 확정 처리.

### 자동 pause 케이스 (모두 슬랙 알림)
1. AI가 답할 수 없는 사실 질문 (facts에 없는 시급 등) → "정책 질문" pause
2. **부분 가능일 답변** — work_hours에 '평일' 포함 시 월~금 중 하루라도 불가 / '주말' 포함 시 토·일 중 하루라도 불가
3. 매니저 수동 일시정지

### Realtime 자동 갱신
- `applicants`(해당 source) 변경 → 목록·뱃지 즉시
- `messages` INSERT → 대화창 즉시 + (열린 후보의) 단계/체크리스트 silent fetch
- `job_candidates` 변경 → stage 배지·체크리스트 즉시

### 응답 텀 (1분)
인입 SMS 시각 + 60초 후에 AI 답장 발송.
- "바로 답장" 위화감 제거
- sleep 중에 같은 후보가 추가 메시지 보내면 **더 늦은 핸들러가 모두 묶어 1회 답변** (coalesce)
- simulate(연습 빙의) / draft 미리보기는 텀 없음

### 마무리 멘트 + 24h 마감 안내
온보딩 진입 시 발송되는 가이드 본문 끝에 자동 부착:
> `⏰ {시각}까지 배민 아이디를 반드시 회신해주세요.`

`{시각}` = 발송시각 + 24h을 **정각으로 반올림**.

## 2-2. 클로드 조련하기

**용도**: AI가 학습할 정보를 한 곳에서. 표시 카테고리 2개.

### 탭 (기본 노출)
1. **운영 정보 (facts)** — AI가 사실로 인용 가능한 운영 정보 (지점·시급·구인 상태·정책)
   - 예) 제목: `강북미아` / 본문: `시급 15,000~20,000원, 토일 08:00-16:00, 픽업 서울 강북구...`
   - 여기 없는 사실은 AI가 추측 금지 → `pause`로 매니저 인계
2. **자동 발송 메시지 (system_message)** — 시스템이 자동 발송하는 고정 문구
   - 키 목록 (제목 = 키. **변경 금지**):
     - `danggeun_start` — 당근/배민 후보 시작 멘트 (둘 다 이거 사용)
     - `apply_received` — apply 폼 접수 안내 (현재 거의 미사용 — 폴백)
     - `baemin_apply_invite` — 배민 triage 후 apply 폼 안내
     - `screening_announce` — 스크리닝 진입 시 안내 묶음 (현재 사용 X)
     - `onboarding_guide` — 앱설치·교육 안내 (마감시각 자동 부착)
     - `onboarding_reminder` — 24h 미회신 시 리마인더
     - `first_day_rules` — 근무 시작 안내
   - 본문에 `{{이름}}`, `{{지점}}`, `{{시간대}}`, `{{지원폼주소}}`, `{{마감시각}}` placeholder 사용 가능

### 비노출 카테고리 (백엔드는 그대로 로드)
- **대화 톤 (conversation)** — UI에서 안 보이지만 `buildToneGuide`가 DB에서 그대로 로드해 AI 프롬프트에 주입
- **스크리닝 운영 문구 (screening)** — deprecated, 미사용

### 편집
- 저장 즉시 **60초 이내** AI 프롬프트에 반영 (캐시 TTL)
- [기본값 채우기] 버튼 — 시드 INSERT (멱등)

## 2-3. 운영

### 대시보드
출근 직후 "오늘 어디까지 와 있나" 파악. 7개 카드(전체/오늘/스크리닝 전/스크리닝 중/스크리닝 완료/확정인력/대기자) + 지점별 표.

### 지원자 목록
모든 지원자 검색·필터·관리 메인 화면. **노션 스타일 인라인 편집** 지원.

- 위쪽: **[+ 지원자 추가]** 버튼 / 지점·상태 드롭다운 / 이름·전화 검색
- 표 컬럼: 성함 / 연락처 / 지점 / 차량 / 시간대 / 시작가능일 / 상태 / 채널 / 지원일
- **행 어디든 클릭 → 우측 상세 패널** (sticky, 420px). 행 단위로 토글
- **이름 클릭 → 메시지 컨택** 기능은 현재 비활성화 (코드에 주석으로 남김)
- 상세 패널: 인라인 편집(상태·확정 슬롯·확정 지점·시작일·이탈 사유) + **[✏️ 전체 편집]** 버튼 (모달, 모든 컬럼)
- **인라인 편집 가능 컬럼**: 지점 / 시작가능일 / 상태 — 셀을 클릭하면 드롭다운/날짜 선택기로 즉시 수정
- **상태 6종**: **스크리닝 전 / 스크리닝 중 / 스크리닝 완료 / 확정인력 / 대기자 / 부적합** (전체 단계와 통일)
- 확정 슬롯 — **복수 선택**(콤마 join). 매트릭스에 모두 카운트됨

### 수기 지원자 등록 (모달)
**[+ 지원자 추가]** 클릭 → 6섹션 폼:
1. 기본 정보 (이름·전화·생년월일·거주지·지원경로)
2. 차량 (보유·면허·차종·본인명의)
3. 희망 (1·2지망·시간대·시작가능일)
4. 상태·확정 (status·필터·확정 지점/슬롯·현재 근무지점·시작일·이탈 사유)
5. 메모·자기소개·경력
6. 마케팅 동의·카카오 친구

- 필수: 이름·전화·1지망
- 기본값: source='manual' / status='스크리닝 전'
- location 입력 시 자동 지오코딩

### 배송원 컨택
지원자와 SMS 양방향 대화. AI 답변 초안 자동 생성.
- 안 읽음 있는 사람 항상 맨 위
- AI가 정보 부족이라 판단 → ⚠️ "AI 응대 불가" + Slack 알림

> 카카오 1:1 답장은 정책상 API 안 됨 — SMS 양방향만.

### 미분류 인박스 (NEW)
SMS Gateway로 들어왔지만 어떤 지원자에도 매칭이 안 되고, AI triage도 확신을 못한 메시지 모음. `classification='pending'`인 메시지들.

- 표: 발신 / 본문 / 들어온 시각 / [배민 지원자로 분류] / [기타로 분류] 버튼
- 매니저가 한 메시지씩 보고 직접 분류
- '배민 지원자로 분류' → 임시 applicants 생성 + apply 폼 URL SMS 발송
- '기타로 분류' → `classification='other'`로 종결

## 2-4. 매트릭스 — 확정 슬롯

# 3. 지원자 상태(6종)와 진행 흐름

상태는 6개로 단순화됨. 자동 갱신과 매니저 수동 설정이 명확히 분리됨.

| 상태 | 진입 | 의미 |
|---|---|---|
| **스크리닝 전** | apply 폼 작성 전 임시 row (배민 SMS triage 결과) / 매니저 수기 등록 디폴트 | 매니저 대기 |
| **스크리닝 중** | apply 폼 작성 직후 (자동) | AI가 7항목 체크리스트 진행 중 |
| **스크리닝 완료** | 스크리닝 7항목 다 채워짐 (자동) | 온보딩(배민 아이디 수집) 진행 또는 매니저 확정 대기 |
| **확정인력** | 매니저 수동 | 슬롯에 카운트됨 |
| **대기자** | 매니저 수동 | 슬롯 sub에 표시 |
| **부적합** | 자동(자격 미달) 또는 매니저 수동 | 더 이상 진행 안 함 |

## 자동 vs 수동 분리

- AI나 시스템 자동 갱신은 **스크리닝 3종(전/중/완료)** 안에서만 일어남
- 매니저가 한 번 **확정인력/대기자/부적합**으로 바꾸면, **시스템이 절대 덮어쓰지 않음**
- 모든 자동 갱신 쿼리에 `.in("status", ["스크리닝 전","스크리닝 중","스크리닝 완료"])` 가드 존재

# 4. AI 응대 단계

후보 1명 = `job_candidates` 1행. 인입 SMS마다 현재 단계 모듈이 호출됨.

## 단계 흐름 (내부)

```
스크리닝(screening) → 온보딩(onboarding) → 확정(active)
       │                  │
       └── paused (자동/수동) ←─ 매니저 [▶ 재개]
       │                  │
       └── abort (부적합)
```

> 화면 표시는 2단계로 단순화: `screening`="스크리닝 중", `onboarding`/`active`="스크리닝 완료".

## 단계별 책임

| 내부 단계 | 진입 시점 | AI가 하는 일 | 시스템 자동 발송 | applicants.status |
|---|---|---|---|---|
| **screening** | 폼 제출 / 매니저 등록 | 7항목 체크리스트 진행. 미확인 항목 묶어 질의 | 시작 멘트(`danggeun_start`)에 안내 묶음 포함 | `스크리닝 중` |
| **onboarding** | 7/7 완료 | 배민 아이디만 수집. 받으면 마무리 멘트 1회 + 슬랙 '준비 완료' 알림 | 앱설치/교육 가이드(`onboarding_guide`) + 24h 마감 안내 자동 부착 | `스크리닝 완료` |
| **active** | 매니저 수동 / 자동 advance(아이디 수신 시) | (자유 대화) | 매니저 수동 진입 시 X. 자동 진입 시 X. | `스크리닝 완료` (매니저가 별도로 `확정인력`/`대기자`로 변경) |

## 자동 처리 규칙

- **스크리닝 진입 시 자동 true**:
  - 시작 멘트에 안내 묶음 포함 → 프로모션(2)·정산(3)·업무시간(6)
  - work_hours에 '주말' 없음(평일만) → 공휴일(4)
  - 공고 vehicle_required=false → 자차 재확인(1)
- **온보딩 진입 시 자동 발송**: 앱설치/교육 가이드 → 앱설치_안내발송됨 자동 true
- **온보딩 → 매니저 인계 이벤트**:
  - 배민 아이디 수신 → 슬랙 *'온보딩 준비 완료'* 알림 → 매니저가 만남장소·확정 처리
  - 24h 미회신 시 cron이 리마인더 SMS 1회 → 추가 3h 미회신 시 슬랙 *'매니저 전화 인계 필요'*

## paused / abort

- **paused**: 자동(사실 부족·부분 가능일·정책 질문) 또는 매니저 수동. Slack 알림 자동. 매니저 [▶ AI 재개] 누를 때까지 AI 침묵
- **abort**: 자차 없음 / 본인명의 불가 등 자격 미달. `status='부적합'` 자동

## 중복 발송 방지

- 자동 발송(안내 묶음/GUIDE/마무리) 직전 3개 outbound와 동일 본문이면 skip
- AI advance 시그널에 reply_text 같이 와도 router가 무시 (transitions가 책임)
- 온보딩 마무리 멘트: 배민 아이디가 처음 채워진 턴에만 1회

# 5. 데이터 흐름 / 트리거

## 인입 SMS — Supabase Database Webhook 경유

```
지원자 SMS → 법인폰(SMS Gateway 안드로이드 앱)
           ↓ REST API 직접 INSERT
        Supabase messages 테이블 (direction='inbound')
           ↓ Supabase Database Webhook
        POST /api/webhooks/supabase-new-message
           │
           ├─ phone 매칭 시도
           │   ├─ 매칭됨 → router.runAgentForCandidate (AI 자동 응대)
           │   └─ 매칭 안 됨:
           │       ├─ 하드 스팸(광고/URL/비휴대폰) → classification='other'
           │       ├─ Haiku triage → is_baemin + conf ≥ 0.7
           │       │   → 임시 applicants 생성 + apply 폼 URL SMS
           │       │   → classification='baemin'
           │       └─ 그 외 → classification='pending' (매니저 인박스)
```

> ⚠️ 옛 `/api/messages/inbound`는 SMS Gateway가 더 이상 호출 안 함. Supabase Webhook이 진입점.

## 라이브 등록 3경로

### 🔵 매니저 수동 등록 (당근 모드)
```
[+ 새 당근 후보] → POST /api/admin/agent/danggeun/start
  ├ applicants INSERT (source='danggeun', status='스크리닝 중')
  ├ job_candidates INSERT (stage='screening', 자동-true 3항목 + 공휴일)
  ├ 시작 멘트 SOLAPI sendSms
  └ messages INSERT (sent_by='danggeun-start')
```

### 🔵 apply 폼 (당근 / 배민 둘 다)
```
지원자 폼 작성 → POST /api/apply
  ├ 배민이고 phone 매칭되는 임시 row 있으면 UPDATE, 아니면 INSERT
  ├ status = '스크리닝 중' (필터 통과) 또는 '부적합'
  ├ job_candidates INSERT (stage='screening')
  └ 시작 멘트 SMS 발송 (sent_by='danggeun-start' 또는 'baemin-start')
```

### 🔵 매니저 수기 등록 (지원자 목록 + 지원자 추가)
```
[+ 지원자 추가] → POST /api/admin/applicants
  └ applicants INSERT만. job_candidates·시작 멘트 자동 발송 X
     (당근·배민 흐름 외 매니저 전용 입력)
```

## 🟢 응답 트리거 (AI 자동 응답)

```
후보 답장 SMS → SMS Gateway → Supabase INSERT → Webhook
  ├ phone으로 applicants 매칭 → job_candidates.agent_stage 확인
  │     paused/abort → skip (매니저 직접 응대)
  │     screening/onboarding/active → router 호출
  └ router.runAgentForCandidate({ received_at })
        ├ 1분 sleep (시작시각 + 60s 목표)
        ├ coalesce 체크 — 내 인입 이후 더 새로운 인입 있으면 양보
        ├ stage 모듈(Claude) → reply_text + transition
        ├ SOLAPI sendSms + messages INSERT (sent_by='agent')
        └ transitions.applyTransition — 자동 발송 + status 갱신 + Slack
```

## 🟡 매니저 직접 발송
```
대화창 입력 → POST /api/admin/messages/send (또는 당근 탭 매니저 발송)
  └ SOLAPI sendSms + messages INSERT (sent_by='danggeun-manual' 또는 '관리자')
```

## 🎚 매니저 수동 단계 변경 (당근/배민 탭 노드 클릭)
```
POST /api/admin/agent/set-stage { applicant_id, target_stage }
  ├ 이전 단계 체크리스트 자동 true
  └ transitions.applyTransition(advance) 그대로 태움
```

## 연습용은 모든 발송이 simulate
- impersonate 라우트가 `source='danggeun_practice'`면 `simulate=true` 전달
- router/transitions 모두 SOLAPI/Slack skip, DB outbound 기록만
- 응답 텀(1분 sleep)도 skip

# 6. 자동 발송·리마인더 (cron)

`vercel.json`의 `crons`로 등록.

## `/api/admin/cron/onboarding-reminder` (매시간 `0 * * * *`)

온보딩 단계 후보를 매시간 점검해 두 단계로 동작:

### 단계 A — 리마인더 SMS
조건: `agent_stage='onboarding'` AND `meta.onboarding_entered_at < now - 24h` AND
`meta.onboarding_reminder_sent_at IS NULL` AND 배민 아이디 미수신
→ `system_message.onboarding_reminder` 본문으로 SMS 1회 발송
→ `meta.onboarding_reminder_sent_at` 기록

### 단계 B — 매니저 전화 인계 슬랙
조건: `meta.onboarding_reminder_sent_at < now - 3h` AND
`meta.manager_handoff_alerted_at IS NULL` AND 여전히 미수신
→ Slack 🚨 *"📞 매니저 전화 인계 필요 — 온보딩 미회신"* 1회
→ `meta.manager_handoff_alerted_at` 기록

> 수신된 후보는 어느 단계도 발동 안 함. 연습용은 슬랙 skip + 메타만 갱신.

# 7. 비용 추적 (AI + SMS)

모든 Claude API 호출과 SOLAPI 발송 비용을 자동 추적.

## AI 토큰 (`ai_usage_daily` + `messages` 컬럼)

- **`messages.model / tokens_in / tokens_out / cache_read_tokens`** — 메시지별 토큰
  - outbound (AI 답장) → INSERT 시 함께 저장
  - inbound (배민 triage 분류) → 분류 후 UPDATE
  - 그 외(매니저 발송·매칭 inbound) → NULL
- **`ai_usage_daily`** — (KST 일자, 모델, 용도) 일별 집계. 비메시지 호출(공고 생성/추출)도 포함
- 용도 7종: `screening` / `onboarding` / `exploration` / `triage` / `job_generate` / `job_extract`

## SMS 발송 비용 (`messages` 컬럼)

- **`messages.sms_type / sms_cost_krw`** — DB 트리거가 outbound INSERT 시 자동 분류·과금
- SOLAPI 단가 기준:
  - `SMS` ≤ 90 바이트 — 18원
  - `LMS` ≤ 2000 바이트 — 45원
  - `MMS` 그 외 — 110원
  - `ALIMTALK` (카카오 알림톡) — 8원

## 통합 view — `usage_daily_cost`

AI(USD → KRW 환산 1400원/$) + SMS 합산. 일별 행:
```sql
SELECT * FROM usage_daily_cost ORDER BY day DESC LIMIT 30;
```
컬럼: day, ai_cost_usd, ai_cost_krw, sms_cost_krw, total_cost_krw, ai_call_count, sms_count, lms_count, alimtalk_count

> 어드민 UI에 비용 카드는 의도적으로 추가 안 함 — 매니저 업무와 무관, 운영자가 필요할 때 SQL로 보면 됨.

# 8. 매니저 일과 / 자주 헷갈리는 점

## 작업 빈도

| 빈도 | 작업 |
|---|---|
| **수시 (실시간)** | 🥕📱 당근/배민 — paused 후보 답변 + [▶ AI 재개], ⏸ 수동 일시정지 |
| **수시** | 미분류 인박스 — pending 메시지 매니저 직접 분류 |
| **공고 등록 시** | 🥕 [+ 새 당근 후보] 또는 ⭐ 추천 받기 |
| **매일** | 대시보드 / 지원자 목록 확인 |
| **주 1회** | 🧠 클로드 조련하기 — 운영 정보(facts) 갱신 |
| **변경 시** | 지점 관리 — 슬롯 정원, 활성 토글 |
| **수기 입력** | 지원자 목록 [+ 지원자 추가] 또는 인라인 편집 |

## 자주 헷갈리는 포인트

- **상태 6종**은 자동(스크리닝 3종)과 수동(확정인력/대기자/부적합) 영역이 분리. 매니저가 한 번 수동으로 바꾸면 시스템이 절대 안 덮어씀
- **시작일은 AI가 안 물음** — "에이전트 질문에 긍정 = 근무 확정"이 아니라는 점을 반영. 매니저가 별도로 처리
- **paused 후 답장만 보내고 [▶ AI 재개] 안 누르면 AI 영영 침묵** — 버튼 꼭 누르기
- **`facts`(운영 정보)에 없는 사실은 AI가 절대 추측 X** → 새 지점 추가 시 facts에도 추가
- **응답 텀 1분** — AI 답장이 60초 후에 도착하는 게 정상
- **여러 메시지 빠르게 보낼 때** — 마지막 인입의 핸들러가 모두 묶어 1회만 답장 (coalesce)
- **부분 가능일 답변** ("수요일은 못해요") → AI 임의 답변 안 함, 즉시 pause + 슬랙
- **연습용 ↔ 라이브 시작 멘트는 한 곳(DB)에서 관리** — 연습에서 다듬으면 라이브에도 그대로 반영
- **지점 활성 토글 OFF는 /apply에만 영향** — admin 모든 화면은 그대로 표시
- **만남장소·차량번호 자동 발송 없음** — 배민 아이디 수신되면 슬랙 '준비 완료' 알림 → 매니저가 직접
- **이름 클릭 메시지 컨택은 비활성화** — 행 클릭 시 우측 상세 패널이 뜸

## Supabase Database Webhook

SMS Gateway(안드로이드 앱)가 Supabase REST API에 직접 INSERT하므로, Supabase Studio에서 webhook 등록 필요:
- Table: `messages`
- Events: INSERT
- HTTP method: POST
- URL: `https://<프로덕션 도메인>/api/webhooks/supabase-new-message`
- Headers: `Authorization: Bearer <SUPABASE_WEBHOOK_SECRET 값>`
